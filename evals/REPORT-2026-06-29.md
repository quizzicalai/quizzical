## 1. Executive summary

The agent makes ~8–25 LLM calls per quiz across seven tunable functions. We built
a top-level `evals/` framework that, for each function, sweeps
`(model x prompt x knob)` variants over a pinned, bucket-balanced input set,
repeats each cell N times to tame non-determinism, and selects a winner by the
product's priority order **cost -> speed -> quality** — with hard quality/latency
floors so "cheapest" can never mean "broken."

This closes launch-audit finding #5 ("the eval harness has zero token/cost
accounting so the team literally cannot measure or optimize spend"): every call
now captures tokens and prices them against a version-pinned snapshot.

**What a live run will tell us (hypotheses the framework is built to test):**
- Can the **loop hotspots** (`decision_maker`, `next_question_generator`) move to a
  cheaper tier (`gemini-2.5-flash-lite`, $0.10/$0.40 per Mtok) without dropping
  below the quality floor? These run 6–12x/quiz, so a per-call cost cut compounds.
- Is the expensive `profile_batch_writer` (6000-token cap on
  `gemini-flash-latest`, the single largest cost line) beaten by `gpt-4o-mini`
  at ~6x lower output price with statistically-equivalent quality?
- Does production's R11 swap to `gpt-4o-mini` on the loop functions still hold up
  under paired significance testing with cost attached (it was chosen on latency
  + a 5-rep bench with no cost/CI)?

## 2. The framework at a glance

```
config/<function>.yaml      datasets/<function>.json
  (model x prompt x knob          (pinned, bucket-balanced
   variants + quality floor        inputs; canonical rosters
   + latency budget)               as ground truth)
        |                                  |
        +--------------+--------------------+
                       v
   runner.py  -- for each (variant x input x rep):
                  render PROD prompt (imported from app.agent.prompts)
                  -> call model (caller.py: MockCaller | LiveCaller)
                  -> capture tokens->$ (pricing.py) + wall latency
                  -> deterministic checks (checks.py)
                  -> LLM-judge quality (judges.py, fixed cross-family judge)
                       v

                       v

                  cost CI . p50/p95 . quality bootstrap CI . validity . checks

                  -> cost->speed->quality decision + Pareto frontier
                       v
              results/report.md   (this file's live-data sibling)
```

Nothing makes a paid call without `--live`. The default `--dry-run` path is fully
deterministic and free, so CI can validate the math.

## 3. How to read every table

- **`$/1k calls`** — mean USD per 1,000 calls (cost/call x 1000, for legibility),
  priced from the 2026-06-29 snapshot in `pricing.PRICE_OVERRIDES`.
- **`p50 s` / `p95 s`** — median and 95th-percentile wall-clock latency. p95 is
  the gate; reasoning models have fat tails the mean hides.
- **`quality (CI)`** — judge agg on a 1–5 Likert scale with a 95% **bootstrap**
  CI (10,000 resamples). **The decision gates on the CI lower bound**, not the
  point mean, so a within-noise edge can't win.
- **`valid`** — fraction of reps that returned schema-valid, parseable output.
- **`checks`** — pass-rate of the deterministic structural gates.
- **Significance tables** — delta-quality is the **paired** mean (variant minus
  incumbent) over identical inputs; "sig?" is the Benjamini-Hochberg (FDR) verdict
  for the exploratory sweep. A directional win also requires the **delta-CI to
  exclude 0** and the paired-t / Wilcoxon p-values to agree.
- **Pareto frontier** — variants not dominated on (cost-down, p95-down,
  quality-up); the automatic pick is always one point on it.

> **Mock-data note.** Some checks below read `0%` (e.g.
> `profiles_cover_all_names`, `planner_matches_canonical`) and one reads `33%`
> (`questions_count_ok`). These are **artifacts of the mock stub outputs**, not
> framework bugs — the stubs emit generic `"Outcome i"` names and a fixed
> question count, so the gates correctly disqualify them. On a live run these
> reflect real model behaviour. They are shown here precisely to demonstrate the
> gates fire.

## 5. The cost -> speed -> quality decision, walked through

Take `next_question_generator` (section 4.1) as the worked example. The rule
(`decision.py`) runs in this exact order:

1. **Eligibility gate.** Keep only variants that (a) return valid output >=95% of
   the time, (b) pass every required check >=98%, (c) have **quality CI-lower >=
   4.0**, and (d) p95 <= 8 s. -> `flash_latest_was` is dropped (p95 9.2 s). Three
   survive.
2. **Cost (priority 1).** Cheapest survivor = `flash_lite_cheap` at $0.445/1k.
3. **Speed (priority 2).** Tie-break among variants within 10% of the cheapest by
   p95. (Only `flash_lite_cheap` is in that band here.)
4. **Quality (priority 3).** Final tie-break by mean judge score. -> Winner
   `flash_lite_cheap`.

Crucially, quality entered as a **floor and a tiebreaker**, never as the primary
objective — that is what "cost-first lexicographic" means. And because we gated on
the **CI lower bound**, we never declared a quality winner that was really just
noise (all the delta-CIs here straddle 0).

**When to override the automatic pick:** consult the Pareto frontier. For
`final_profile_writer`, if a live run shows a *significant* quality gain for a
pricier model (delta-CI excludes 0, survives Holm correction), a human may pick
the spend-up point — the framework surfaces it but, per priority order, won't
choose it automatically unless the cheaper option fails the floor.

**When the framework refuses to choose:** if no variant clears the floor (several
functions show `FLOOR NOT MET` above, driven by mock stubs), it returns the
best-quality option **with a do-not-ship warning** rather than silently shipping a
sub-floor config — the same "strict improvement, no silent downgrade" discipline
the precompute promotion gate uses.

## 6. What must be run to get real numbers

| step | command | cost | time |
|---|---|---|---|
| budget check (no calls) | `cli plan --reps 30` | $0 | instant |
| variance pilot | `cli run --live --reps 5` | ~$2 | ~5 min |
| full sweep (size N via power analysis) | `cli run --live --reps 20 --concurrency 6` | ~$9–13 | ~50–60 min |
| rebuild report | `cli report` | $0 | instant |

Run the live sweep **under the backend venv** (`backend/.venv312` or the project
`.venv`) so prompts are imported from `app.agent.prompts` rather than the built-in
fallbacks, and set `OPENAI_API_KEY` / `GEMINI_API_KEY`. Use the `--reps 5` pilot
to estimate sigma-delta, then `stats.min_reps_for_effect(sd_delta, mde=0.15)` to
size the full run (rule of thumb: ~10–20 reps/input; bias the loop hotspots
higher).

**Per-quiz cost projection.** Multiply each function's winning per-call cost by
its calls/quiz (section 2) and sum: with a 6–12-question quiz the loop functions
(`decision_maker` + `next_question_generator`, ~7x each) dominate, so a successful
move of *those two* to a cheaper eligible tier is the highest-leverage change.
`report.py` prints the summed per-call cost; the live report adds a $/quiz line
once real token counts replace the mock proxy.

### Statistics — checked for real errors, found none material

| Claim | Verified in code | Verdict |
|---|---|---|
| Bootstrap CI, 10k, percentile, fixed seed | `stats.bootstrap_ci` resamples *with replacement* at the correct n, sorts, takes 2.5/97.5 percentiles, seeded. | Correct |
| Paired tests run on **paired** data | `report._paired_quality_vectors` aligns the two variants by `(input_id, rep)` and intersects keys *before* `paired_compare`; `paired_compare` raises on length mismatch and tests the delta vector. | Correct — genuinely paired, not just labelled so |
| Paired-t + Wilcoxon backstop | `ttest_rel` via scipy; Wilcoxon on non-zero deltas, suppressed when <6 pairs (sensible — Wilcoxon is unreliable below that). | Correct |
| Holm (FWER) | step-down: k-th smallest p vs `α/(m−rank)` (0-indexed `m−rank` == `m−k+1`), `break` on first non-reject. | Textbook-correct |
| BH (FDR) | step-up: find largest rank with `p ≤ (k/m)·α`, reject all up to it. | Textbook-correct |
| CI-lower-bound is the floor gate | `decision._eligibility_reason` gates on `v.quality.lo < quality_floor`, **not** the mean. Unit test `test_decision_gates_on_ci_lower_bound` proves a high-mean/low-CI-lower variant is rejected. | Correct, and genuinely conservative — this is the right choice |
| Cost→speed→quality lexicographic | `decision.decide`: eligibility → cheapest mean $ → cost band → fastest p95 → max quality. Unit tests confirm cost-first, latency-budget rejection, floor-not-met fallback. | Correct, matches the user's stated priority order |
| Power / MDE sizing | `n = ((z_{1−α/2}+z_{1−β})·σ/MDE)²`, paired normal approx; inverse provided. | Correct standard formula |
| Price snapshot reproducible | pinned `PRICE_OVERRIDES` → litellm → 0.0-with-warning; bills against `completion_tokens` (reasoning folded in for billing, kept separately for diagnostics — the right call). | Correct |

**Real (but minor) findings — none invalidate a conclusion:**

1. **Holm is documented for the final go/no-go but not auto-applied in
   `report.py`.** `build_function_report` corrects the *exploratory* family with
   **BH only**; `holm_bonferroni` exists, is unit-tested, and is described as the
   gate for "the chosen config," but the report pipeline never calls it. The
   decision rule (`decide`) gates on the quality **CI lower bound** instead, which
   is actually a *stricter* per-variant guard than a Holm-corrected p-value — so
   the final call is not under-protected. Still, the methodology's "Holm for the
   final decision" is currently a *described* step, not a *wired* one. Flag:
   documentation slightly over-claims the pipeline's current behavior. Easy fix:
   apply `holm_bonferroni` to the single incumbent-vs-winner comparison at
   decision time, or soften §3.4 to "Holm is provided for the final confirmatory
   test" (which the harness supports but the auto-report doesn't run).

2. **Undocumented 5% p95 speed-band slack.** `decide` keeps variants within
   `cost_tie_pct` (10%, documented) of cheapest, then within **`fastest*1.05`**
   p95 before the quality tiebreak. That 1.05 is a reasonable "don't let trivial
   p95 noise decide" guard, but it's a hard-coded magic number not mentioned in
   `methodology.md` §4. Cosmetic; document it.

3. **BH p-value family uses only the paired-t p, not the Wilcoxon.** `report.py`
   feeds `pt.t_pvalue` into BH; the Wilcoxon p is displayed but not corrected or
   combined. The methodology says "if the two disagree the effect is fragile"
   (a human-read backstop), so this is consistent with the stated design — just
   note the FDR control is on the t-test family specifically.

4. **`benjamini_hochberg` assumes independent/PRDS p-values** (standard BH, not
   BY). Pairwise-vs-one-incumbent comparisons are positively correlated (shared
   incumbent), which BH tolerates under PRDS, so this is fine — but it's an
   assumption worth one sentence in the threats-to-validity section.

5. **Non-statistical: the t-interval fallback `_t_ppf`** (used only when scipy is
   absent) is a Cornish-Fisher-style approximation, not exact. scipy *is* present
   in the target venv so this path isn't exercised here; acceptable as a
   degrade-gracefully fallback, but CIs computed on a scipy-less box would be
   slightly approximate. Already implicitly acknowledged by "scipy is used when
   available."
