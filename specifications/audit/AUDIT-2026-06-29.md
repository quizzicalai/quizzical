# Quizzical / Quafel — Comprehensive Codebase Re-Audit

**Date:** 2026-06-29
**Scope:** Full repo (FastAPI + LangGraph backend, React 19 + Vite frontend, precompute subsystem, `Analysis/` tuning harness, infra) — a **follow-up** to [AUDIT-2026-06-28.md](AUDIT-2026-06-28.md), run after the `hardening/audit-2026-06-28` branch and the subsequent 12 PRs (durable jobs / crash recovery, cluster LLM gate, nightly safety judge, per-result OG meta, funnel analytics, PWA manifest, replica-autoscale IaC) landed.
**Method:** 11 dimension auditors (Security, Reliability, Performance, Agent O&E, UX, Scalability, Architecture, Testing, Virality, Mobile/A11y, Self-learning) re-read the current source with a disprove-first mandate, explicitly checking each prior finding's `resolved / open / new / regressed` status against the code as it stands today.

> Severity: **P0** = launch-blocker / exploitable / unbounded cost / data loss. **P1** = real bug or gap that bites under real traffic. **P2** = correctness edge case, quality, polish, or missing test.
> Status: **resolved** = the prior defect is gone. **open** = a prior finding still stands. **new** = first surfaced in this pass (often a consequence of code added by the remediation). **regressed** = previously partially-fixed, now actively worse than before.

## Executive summary & current posture

**The four P0 launch-blockers from 2026-06-28 are fixed and verified. There are zero open P0s.** The hardening branch did exactly what it set out to do on the cost/abuse/infra front: secrets are out of the image, production now resolves its environment authoritatively (`azure` fails closed through `is_production()` / `NON_PROD_ENVS`), the spoofable left-most `X-Forwarded-For` hop is replaced by a trusted-proxy-aware resolver, and the paid `/quiz/proceed` + `/quiz/next` path is now bounded by a per-`quiz_id` action cap plus a cluster-wide daily start ceiling. Several P1s were also genuinely closed: the max-question forced-finish loop, the status-poll write-amplification, the dead topic-knowledge classifier call, free-text prompt-injection (option-index now required), TrustedHost in prod, the weak-secret fail-closed guard, the Turnstile fail-closed startup assertion, and the iOS 16px input-zoom fix. The nightly promotion path now runs a real, fail-closed two-judge safety/quality gate before signing user content — the single most important self-learning gap from the prior audit. A first-party, privacy-respecting funnel-analytics layer now exists.

**But the verdict is unchanged in substance: still not ready to launch to thousands.** The fixes that landed were the surgical, low-risk ones; the structural tier the remediation explicitly deferred is exactly where the remaining risk concentrates, and the new subsystems introduced their own defects. Three themes dominate:

1. **The durable-jobs / crash-recovery subsystem — the centerpiece of the reliability remediation — has a load-bearing hole and is effectively untested.** The recovery contract depends on a heartbeat that is **never emitted** (`QuizJobRepository.heartbeat()` has zero callers), so any legitimately-slow-but-alive agent run past `stale_after_s` (180 s) is mis-classified as stale and **re-run concurrently**, double-spending paid LLM + FAL. The `claim_stale` sweep is **not provably atomic** across the (up to 5) autoscaled replicas — it lacks `FOR UPDATE SKIP LOCKED` — so two sweepers can double-claim the same job. The recovery re-run does **not** short-circuit an already-completed quiz, so it re-finalizes finished sessions with fresh paid calls and silently overwrites the result the user already saw. The durable `quiz_jobs` row is created **inside** the background task, leaving the highest-probability crash window (right after the 202, before the first DB write) unprotected. And `get_quiz_status` **never reads `quiz_jobs`**, so a deterministically-failed run still polls "processing" until the FE's 60 s timeout. None of `run_agent_in_background`, `_recover_one`, or the cost guards are exercised end-to-end — the integration `_FakeRedis` lacks `incr`/`expire`/`eval`, so the new caps silently fail-open in every test.

2. **The "multi-replica ready" scale story is unverified at the infra layer and disabled in config.** Horizontal scale-out now hinges on a durable Redis checkpointer, but `AsyncRedisSaver` requires RedisJSON/RediSearch modules that **Azure Cache for Redis does not ship**, and on init failure the code silently falls back to a per-process `InMemorySaver` with no fail-closed assertion — so prod can silently run non-durable, non-shared state. Meanwhile `WEB_CONCURRENCY` still defaults to 1, the cluster LLM-concurrency gate is implemented but **`enabled=False`** and turned on nowhere, and the autoscale policy in `main.bicep` is an **advisory output only** — the deploy pipeline never applies a single `--scale-rule`. So the effective LLM ceiling is still `16 × workers × replicas` and the codified scale policy is documentation, not infrastructure.

3. **The virality loop is still broken on the path crawlers actually hit, and the share assets regressed.** The correct per-result OG/SSR meta endpoint was built — but nothing routes `/result/{id}` (or crawlers) to it, *and* the SWA rewrite that does exist targets an origin with no backend, so the endpoint is doubly unreachable in prod. `VITE_PUBLIC_URL` is unset, so `og:image` stays relative. Worst of all, the PWA manifest now ships referencing four icon files (and `og-image.png`, favicon, apple-touch-icon) that **still 404** — a regression from "install silently fails" to "install succeeds with a broken icon, and every fallback share card points at a 404 image."

The good news remains: this is a mature, security-conscious codebase, and the P0 cost/abuse surface is now genuinely defended. The recurring failure mode has shifted from "mechanisms keyed off the wrong env name" to **"mechanisms that are built but not wired, not enabled, or not provisioned"** — the heartbeat with no caller, the cluster gate with `enabled=False`, the autoscale output nothing consumes, the meta endpoint nothing routes to, the judge column nothing writes. Most remaining P1s are still connect-the-existing-pieces fixes, but several (durable-job atomicity/idempotency, a real Redis-modules checkpointer, a dollar-denominated cost ceiling) require care, not just wiring.

## Cross-cutting themes
*(Root causes; the individual findings below are downstream of them. Cross-dimension duplicates are de-duplicated here and cross-referenced in the per-dimension sections.)*

**A. The durable-jobs recovery loop is structurally unsound and re-spends paid LLM/FAL.**
A single subsystem (commit `89c07d0`) underpins reliability, scalability, architecture, and the self-learning recovery path, and the same four defects surface in all of them: (1) `heartbeat()` is dead code so `stale_after_s` is a hard wall-clock deadline, not a liveness signal; (2) `claim_stale` lacks `FOR UPDATE SKIP LOCKED`; (3) `_recover_one` / `_decide_or_finish_node` have no `final_result` short-circuit so finished quizzes get re-finalized and overwritten; (4) the `quiz_jobs` row is created too late and `get_quiz_status` never reads it. Net effect: duplicate concurrent agent runs racing each other's Redis/DB writes and double-charging the #1 cost risk, plus silent mutation of a result the user already saw.
_Spans:_ Reliability, Scalability, Architecture, Self-learning, Testing. Files: `backend/app/services/database.py:525-570`; `backend/app/services/agent_recovery.py:27-65`; `backend/app/api/endpoints/quiz.py:611-689,1456,1613,1793-1800`; `backend/app/agent/graph.py:779-861,929-938`; `backend/app/core/config.py:343-350`.

**B. The "multi-replica ready" story is built but neither provisioned nor enabled.**
Scale-out depends on (i) a durable, module-enabled Redis for the LangGraph checkpointer, (ii) `WEB_CONCURRENCY>1`, (iii) an applied Container-App autoscale rule, and (iv) the cluster LLM-concurrency gate. None of the four is actually live: Azure Cache for Redis lacks RedisJSON/RediSearch (silent `InMemorySaver` fallback, no fail-closed assertion); `WEB_CONCURRENCY` defaults to 1 and the Dockerfile comment that discourages raising it is stale; `apiAutoscalePolicy` is a bicep *output* the deploy never consumes; and `llm.global_concurrency.enabled` is `False` everywhere. The mechanisms exist; the deployment is still effectively single-worker, single-policy-less, per-process.
_Spans:_ Scalability, Architecture, Reliability. Files: `backend/app/agent/graph.py:1096-1153`; `backend/Dockerfile:104-128`; `infrastructure/main.bicep:14-79`; `.github/workflows/api-deploy.yml:374-385`; `backend/app/services/llm_concurrency.py:112-260`; `backend/app/core/config.py:197`; `infrastructure/config-data/base-config.yml:121-122`.

**C. There is still no dollar-denominated cost measurement — the "cost ceiling" counts quiz-starts, not dollars.**
For a product whose stated #1 risk is paid-LLM/FAL cost abuse, no live or offline path captures `resp.usage` or `litellm.completion_cost`. The live "circuit breaker" is a **count** of `/quiz/start` calls (5000/UTC-day), blind to `/proceed`/`/next` agent invocations and to FAL spend; the eval harness records quality and latency but no tokens or dollars; the precompute cost figure is a static per-call estimate (and `cost_cents` can overflow `SmallInteger`). The team cannot attribute spend, compare model cost-per-quiz, detect a cost regression after a prompt/model swap, or set a real budget. (Bonus failure mode: the single global counter is itself a DoS lever — a CAPTCHA farm can exhaust the day's budget and 503 everyone.)
_Spans:_ Agent O&E, Scalability, Security. Files: `backend/app/services/llm_service.py:818-852`; `backend/app/api/endpoints/quiz.py:1103-1133`; `backend/app/core/config.py:321-334`; `backend/Analysis/llm_caller.py:46-51,188-192`; `backend/scripts/_precompute_spend.py:19-21,83-94`; `backend/app/models/db.py:530-531,585-587`.

**D. The self-improving quality loop is now half-wired: the nightly judge is real, but drift gating, the golden set, and the server-side judge column are still dead, and the one judge isn't independent.**
The nightly promotion now runs a genuine fail-closed two-judge consensus before signing user content (the prior P0/P1 — resolved). But the "two judges" are the **same** Gemini-flash model reseeded twice (no independent rater, no human calibration), the offline divergence threshold of 2 points on a 0–100 scale likely over-rejects (near-no-op loop), `run_drift_probe`/`evaluate_golden`/`strictly_improves` still have **zero callers**, `evaluator_drift_paused` is never persisted or read, and `judge_plan_score` is **still never written** so the server-side `min_judge_score` gate is a NULL-passes no-op.
_Spans:_ Agent O&E, Self-learning, Architecture. Files: `backend/scripts/promote_user_quizzes.py:140-299`; `backend/scripts/_precompute_judge.py:30,112-161`; `backend/app/services/precompute/evaluator.py:100-121`; `backend/app/jobs/evaluator_drift_probe.py:30-55`; `backend/app/services/precompute/golden.py:30-55`; `backend/app/api/endpoints/admin_precompute.py:489-491`.

**E. The viral surfaces are built but unreachable, and the brand assets regressed.**
The growth thesis ("I'm <X> — find out yours!" with a unique AI image) still collapses to one generic card on the path crawlers see. The per-result meta endpoint exists but nothing routes `/result/{id}` to it; the SWA `/result-meta/*` rewrite targets an origin with no backend; `VITE_PUBLIC_URL` is unset so `og:image` stays a relative path crawlers ignore; and the manifest now references icons (plus `og-image.png`, favicon, apple-touch-icon) that 404 — a regression. Funnel analytics ships but omits the top-of-funnel (no `landing_view` / synopsis-advance), so the largest drop-off (land → start) is still invisible.
_Spans:_ Virality, UX, Mobile/A11y, Security, Testing. Files: `frontend/src/pages/FinalPage.tsx:207`; `frontend/staticwebapp.config.json:2-11`; `frontend/public/manifest.webmanifest:10-37`; `frontend/index.html:18-21,30-35`; `frontend/vite.config.ts:23-40`; `backend/app/api/endpoints/results.py:45,53-86`; `.github/workflows/azure-static-web-apps-*.yml`.

**F. Image-pipeline DB fan-out and missing `pool_timeout` can still exhaust the pool under concurrent quizzes.**
`generate_character_images` opens 2–3 DB sessions **per character** concurrently (10–18 checkouts for one 5-character set) on top of the background runner's own sessions, against a `pool_size=20 + max_overflow=10` pool with **no `pool_timeout`** — so a saturation event makes the hot `/status` rehydrate and `/media` reads stall up to SQLAlchemy's 30 s default before erroring. The prior batching fix was reverted (`2dae8cb`). Cross-topic character-image bleed (same name → wrong image) is part of the same un-batched lookup path and remains user-visible on the core feature.
_Spans:_ Performance, Scalability, Reliability. Files: `backend/app/services/image_pipeline.py:302-326,385-436`; `backend/app/api/dependencies.py:64-73`.

**G. CI's quality gates are inert: the coverage gate measures nothing and flaky tests can red the deploy.**
The 85% coverage `fail_under` never fires because CI runs pytest from `backend/` where `--cov=backend/app` collects zero data and exits 0 (measured: 28% when it measures at all). Two timing/nondeterministic flakes (cluster-cap wall-clock race; random topic-chip draw) can red the deploy under `--maxfail=1`. Test tooling is version-skewed (pytest 9 local vs `<9` pinned). So new code can ship with arbitrarily low coverage and a green-or-red outcome that depends on luck.
_Spans:_ Testing. Files: `pytest.ini:21-24`; `.coveragerc:6-7`; `.github/workflows/api-deploy.yml:136,174-175,189-193`; `backend/tests/unit/services/test_llm_concurrency_cluster.py:202-220`; `frontend/src/components/landing/TopicSuggestionExplorer.spec.tsx:15-27`.

## Per-dimension findings (open / new / regressed)

### Security

#### [P1 · open] No content-moderation gate before the paid live pipeline (`safety_checker` is still dead code)
- **Location:** `backend/app/agent/graph.py:405-503` (`_bootstrap_node`); `prompts.py:312`; `schemas.py:160,709,742`; `quiz.py:1142-1175` (`start_quiz`)
- **Problem.** The `safety_checker` prompt + `SafetyCheck` schema still exist but are never invoked on the live free-text path (callers are only the admin/precompute DB policy gate, not a content classifier). `_bootstrap_node` feeds the raw user category into the paid LLM + FAL pipeline with no blocklist/moderation, and the output is served via the unauthenticated `GET /result/{id}` and now amplified by the `/result-meta` crawler card. The nightly promotion judge gates *packs*, not the live request path.
- **Fix.** Add a fail-closed moderation gate in `start_quiz` **before** any paid work: a static blocklist/regex over the normalized category (fail-closed for clearly-disallowed terms), optionally backed by a single cheap moderation call wiring the existing `safety_checker`. Short-circuit unsafe topics with 422 and never enter the graph or schedule image jobs; apply on the request-path dependency so background and pack-miss paths are covered.

#### [P1 · open] Operator "2FA" is still a presence-only header check, not a real second factor
- **Location:** `backend/app/api/dependencies.py:350-387` (`require_operator`)
- **Problem.** Part (a) of the prior finding is resolved (the gate now fires in prod via `is_production`). Part (b) is open: `two_factor = request.headers.get('x-operator-2fa','').strip()` and `two_factor_verified = bool(two_factor)` — any holder of `OPERATOR_TOKEN` passes by sending `X-Operator-2FA: x`. There is no TOTP/HMAC validation, so admin mutating endpoints (enqueue/promote/rollback/GDPR-forget/import) are protected by a single static bearer token. Impact is reduced from the prior audit because the bearer is now strong (fail-closed ≥32 bytes) and the gate executes.
- **Fix.** Either implement a genuine second factor (validate `X-Operator-2FA` as a TOTP code or HMAC challenge against a server-held per-operator secret with `hmac.compare_digest`, setting `two_factor_verified` only on a cryptographic match), or drop the "2FA" wording to avoid false assurance and document the single-bearer model.

#### [P1 · open] Per-IP `/quiz/start` budget remains generous; cost-tiering of the global bucket not implemented
- **Location:** `backend/app/core/config.py:362-380`; `backend/app/security/rate_limit.py:83-88` (`bucket_key`)
- **Problem.** `quiz_start_rate_limit` is still capacity=3, refill 1/30 s (~120 starts/hr from one non-spoofed IP, each fanning out synopsis + character LLM calls and multiple FAL jobs). The global middleware bucket still keys on the coarse `/api/quiz` prefix, so cheap `/status` polling shares the same budget as expensive `/start` and `/next`. The `LiveCostGuard` daily ceiling materially reduces worst-case dollar exposure, so this is now tuning/scale, not a launch blocker.
- **Fix.** Lower `quiz_start_rate_limit` toward a human rate (e.g. capacity=2, refill 1/120–300 s) and split `bucket_key()` by cost tier so cost-bearing POSTs (`/start`, `/proceed`, `/next`) get a tighter bucket than cheap GET polling.

#### [P2 · new] Turnstile `remoteip` still uses the spoofable left-most XFF hop
- **Location:** `backend/app/api/dependencies.py:200-218` (`_client_ip_for_remoteip`), 255-257
- **Problem.** The rate-limit `_client_ip` was hardened to the trusted right-most hop, but the Turnstile siteverify `remoteip` field is still `xff.split(',')[0]`. Cloudflare binds a token to a client IP for replay defense; sending an attacker-chosen IP lets a token captured via XSS/MitM be replayed from another network by forging the matching XFF, partially defeating that defense-in-depth layer — inconsistent with the centralized fix.
- **Fix.** Reuse the centralized `app.security.rate_limit._client_ip` for `remoteip`, or omit `remoteip` when the trusted hop cannot be resolved.

#### [P2 · new] Global daily cost ceiling is also a denial-of-service lever
- **Location:** `backend/app/api/endpoints/quiz.py:1103-1133`; `backend/app/core/config.py:321-334`
- **Problem.** `_enforce_global_daily_cost_ceiling` increments a single cluster-wide counter on every Turnstile-passed `/quiz/start` and 503s **all** users for the rest of the UTC day once `max_quiz_starts_per_day` (5000) is hit. An adversary with Turnstile-solving capacity (a CAPTCHA farm) can intentionally exhaust the global budget to take the product offline for everyone — converting the cost backstop into an availability attack. This is the inherent tradeoff of a single global counter and was the explicitly-requested design.
- **Fix.** Accept as a deliberate cost-over-availability tradeoff for launch, but add: alerting/auto-raise on rapid counter climb, a per-IP daily sub-cap so one source cannot consume a large share, and an operator override flag to lift the cap without a redeploy.

#### [P2 · new] `/result-meta` builds canonical/`og:url` from the request Host when `PUBLIC_SITE_URL` is unset
- **Location:** `backend/app/api/endpoints/results.py:53-69` (`_public_base_url`), 205-207
- **Problem.** `PUBLIC_SITE_URL` is unset in IaC/workflow, so `_public_base_url` falls back to `request.base_url` (client-supplied Host). TrustedHost constrains the Host to the prod allowlist, so blast radius is limited, but a crawler card's `og:url`/canonical could point at the API/ingress host rather than the FE domain, and any future `TRUSTED_HOSTS` loosening widens this. The HTML body is fully escaped, so there is no markup injection.
- **Fix.** Set `PUBLIC_SITE_URL=https://quafel.com` (the canonical FE origin) in the deployment so crawler cards always advertise the real front-end domain. *(Same root cause as the virality `VITE_PUBLIC_URL` P1.)*

### Reliability
*(Theme A — durable-jobs recovery — is the dominant reliability cluster; see also its Scalability/Architecture/Self-learning duplicates, de-duplicated in the punch list.)*

#### [P1 · new] Durable `quiz_jobs` row is created only INSIDE the background task, leaving the highest-probability crash window unprotected
- **Location:** `backend/app/api/endpoints/quiz.py:1456` (`/proceed add_task`), 1613 (`/next add_task`), 639-640 (`mark_running` inside `run_agent_in_background`)
- **Problem.** `/proceed` and `/next` schedule `run_agent_in_background` **without** first creating a `quiz_jobs` row; the row is created only when the bg task calls `_quiz_job_update(..., "running")` — which runs *after* the 202 is returned. If the worker is killed in the window between scheduling and that first DB write (deploy / OOM / ACA scale-in / SIGTERM), **no `quiz_jobs` row ever exists**, so the recovery sweeper (which claims only `status='running'`) has nothing to claim: the quiz is permanently stranded in "processing" and the paid work is lost — exactly the failure mode the durable-jobs system exists to prevent, at the single most likely kill instant.
- **Fix.** Create the `quiz_jobs` row synchronously in the handler (inside the held session lock for `/next`) via `QuizJobRepository.mark_running(quiz_id)` + commit **before** scheduling; `run_agent_in_background`'s `mark_running` then becomes a heartbeat/attempt-bump on an existing row.

#### [P1 · new] `claim_stale` lacks `FOR UPDATE SKIP LOCKED` — not reliably atomic across replicas (`apiMaxReplicas=5`)
- **Location:** `backend/app/services/database.py:546-570`; `infrastructure/main.bicep:40`; `backend/app/services/agent_recovery.py:8-9`
- **Problem.** `claim_stale` does `UPDATE … WHERE quiz_id IN (SELECT … LIMIT n) RETURNING` with no `FOR UPDATE SKIP LOCKED` on the subquery. Under READ COMMITTED with a hash/materialized semi-join plan, two concurrent sweepers (the loop runs in every replica, up to 5) can both materialize the same `quiz_id`; whether the second UPDATE drops it depends on EvalPlanQual re-evaluation, which is plan-dependent and not guaranteed → the **same job claimed and re-run by two replicas**, double-spending paid LLM/FAL. The docstring asserts multi-replica safety, but the canonical safe pattern is absent and there is no concurrency test.
- **Fix.** Make the claim atomic: `SELECT quiz_id … FOR UPDATE SKIP LOCKED LIMIT n` (SQLAlchemy `.with_for_update(skip_locked=True)`), then UPDATE those ids with `RETURNING`; keep a portable branch for sqlite tests. Add an integration test firing two concurrent `claim_stale()` against the same stale row and asserting exactly one wins.

#### [P2 · open] Session recovery fetch has no AbortSignal and a non-atomic guard

#### [P1 · new] Crash-recovery sweeper re-runs already-completed quizzes (duplicate paid final-result + image, overwrites the user's result)
- **Location:** `backend/app/services/agent_recovery.py:27-65`; `backend/app/api/endpoints/quiz.py:590-608,675-687`; `backend/app/agent/graph.py:780-861,929-938`
- **Problem.** The sweeper claims any `quiz_jobs` row stuck `status='running'` with a stale heartbeat and unconditionally calls `run_agent_in_background`. The job flips to "succeeded" only inside that task's `finally` via `_quiz_job_update`, which **swallows all errors** — so a run that completed (final_result persisted) but whose `mark_succeeded` write failed (pool exhaustion / transient reset) leaves the row "running." With no mid-run heartbeat, the row goes stale and the sweeper re-claims it; `_recover_one` loads the completed state (Redis still has `final_result`, or DB rehydrate rebuilds it) and re-enters the graph. `_phase_router` routes the completed state to `decide_or_finish`, which has **no `final_result` guard**: it makes a fresh paid decision call and, on FINISH_NOW, re-runs the paid `tool_write_final_user_profile`; `_persist_adaptive_and_final` then `mark_completed`-overwrites the result and re-runs the paid FAL result image with no `is_completed` check. Net: duplicate paid LLM+image spend and silent mutation of the result the user already saw. *(Theme A; the Reliability "recovery re-run does not short-circuit" P2 is the same root cause, escalated here to P1 because of the cost + overwrite.)*
- **Fix.** Make recovery idempotent for finished work: short-circuit `decide_or_finish` when `state.get('final_result')` is populated (return `should_finalize=True` with no LLM/image call); in `_recover_one`, skip re-running and mark "succeeded" when the durable `SessionHistory` row is already `is_completed=True`; emit periodic `heartbeat()` calls during long runs.
