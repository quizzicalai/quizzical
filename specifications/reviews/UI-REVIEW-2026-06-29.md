# Quizzical / Quafel ‚Äî UI Modernization Punchlist (Visual / CSS only)

**Date:** 2026-06-29
**Scope:** Frontend at `frontend/` (React 19 + Vite). Purely **visual / CSS-level** modernization ‚Äî color, spacing, typography, borders, radii, elevation/shadow, motion, focus. **No UX change**: same flows, same screens, same content, same component behavior and information shown.
**Goal:** more *modern* and more *minimal*. Never busier.

---

## 1. Summary

The app is already minimal and reasonably well built: it uses an **RGB-channel CSS-variable design-token system** (`--color-*` defined in `src/index.css :root`, surfaced as Tailwind semantic colors via `tailwind.config.js`'s `withOpacity()` helper), plus a parallel set of **landing-page layout tokens** (`--lp-*`) and **typography tokens** (`--font-*`). It already respects `prefers-reduced-motion` globally, ships a 44px minimum tap target on key controls, and uses focus-visible rings throughout.

The biggest opportunities are not "add more" ‚Äî they are **regularize and tighten**:

1. **Spacing is ad-hoc, not on a grid.** Values like `0.34rem`, `0.72rem`, `0.74rem`, `1.05rem`, `0.28rem`, `1.95rem`, `0.875rem`, `mt-7` are scattered through `index.css` and components. None of these are a clean multiple of 4/8px. A modern minimal UI gets most of its calm from a **strict spacing scale** (4/8pt). Normalizing to an 8pt grid (4px half-steps allowed) is the single highest-impact, lowest-risk change.
2. **Typography is not on a modular scale and has no shared type tokens.** Sizes are a mix of `clamp()`, raw `rem`, Tailwind `text-xs‚Ä¶text-4xl`, and arbitrary `text-[10px]`/`text-[14px]`/`text-[12px]`. A documented modular scale (1.25 "major third") expressed as `--font-size-*` tokens gives instant polish and consistency.
3. **Heavy, multi-layer card shadow.** `.lp-card` uses a 4-layer shadow topping out at `0 42px 86px -26px`. That is a strong, almost dramatic elevation for a minimalist product. 2025 guidance is *barely-there* layered depth. Softening this reads as more modern and more minimal at once.
4. **Two contrast/accessibility risks** (muted text on tinted card; the disabled submit button) are worth fixing explicitly under WCAG 2.2.
5. **Radii are inconsistent** (`0.875rem`, `1.25rem`, `rounded-md`, `rounded-xl`, `rounded-2xl`, `9999px`) with no token scale. A small radius token scale tidies this.

Because nearly all styling already flows through CSS variables, **most of this punchlist is centrally applyable**: add/normalize tokens in `:root` (and a few in `tailwind.config.js`), and the components inherit the change. The component-level arbitrary values (`text-[10px]`, `p-4`, `mt-7`, `0.34rem`) are the exceptions that need per-file edits.

---

## 2. Tailwind or plain CSS vars? (how to apply changes)

**Both, layered ‚Äî and that is good news.**

- **Tailwind** is the utility engine (`tailwind.config.js`, `postcss.config.js`, `@tailwind base/components/utilities` in `src/index.css`). Components use Tailwind utility classes (`p-4`, `rounded-2xl`, `text-fg`, `shadow-sm`, `bg-primary`, etc.).
- **Plain CSS custom properties** are the source of truth for color/typography/landing-layout. `tailwind.config.js` maps semantic color names ‚Üí `rgb(var(--color-x, <fallback>) / <alpha>)` via `withOpacity()`. So `bg-primary`, `text-muted`, `ring-ring/60` etc. all resolve to a CSS var with a baked numeric fallback.

**Implication for applying this punchlist:**
- **Color, type-size, landing-layout tokens** ‚Üí change centrally in `src/index.css :root` (and keep the numeric fallbacks in `tailwind.config.js`'s `FALLBACK` map in sync ‚Äî they are duplicated by design).
- **Radius / shadow / spacing scale** ‚Üí Tailwind's `theme.extend` does **not** currently define custom radius/shadow/spacing; components use Tailwind's built-in scale or arbitrary values. To tokenize these, either (a) add `borderRadius`, `boxShadow`, `spacing` keys to `theme.extend` and swap utilities, or (b) define `--radius-*` / `--shadow-*` / `--space-*` CSS vars and reference them in the hand-written `.lp-*` rules. Option (b) is lower-risk for the `index.css` rules; option (a) is cleaner for the component utilities. Either keeps changes central.

---

## 3. Highest-impact recommendations (do these first)

1. **Adopt a strict 8pt spacing scale** and snap the ~20 off-grid values in `index.css`/components to it. (Section: Spacing & layout) ‚Äî biggest calm-per-effort win.
2. **Soften `.lp-card` elevation** from the 4-layer dramatic shadow to a 2-layer barely-there shadow. (Elevation)
3. **Introduce a modular type scale as `--font-size-*` tokens** (1.25 ratio) and replace arbitrary `text-[10px]`/`text-[12px]`/`text-[14px]` + the raw `0.68rem`/`0.74rem` chip sizes. (Typography)
4. **Fix the two WCAG 2.2 contrast risks** (muted body text alpha; disabled submit label). (Color & contrast)
5. **Standardize a radius token scale** and collapse the 6 different radii to ~4 steps. (Borders & radii)

## 5. Centralization plan (so changes apply globally)
