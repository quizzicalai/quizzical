#### P1 — Crash-recovery sweeper re-runs already-completed quizzes (duplicate paid final-result + image, overwrites the user's result)
- **Where:** backend/app/services/agent_recovery.py:27-65; backend/app/api/endpoints/quiz.py:590-608,675-687; backend/app/agent/graph.py:780-861,929-938
- **Problem:** The recovery sweeper claims any quiz_jobs row stuck status='running' with a stale heartbeat and unconditionally calls run_agent_in_background. The job is only flipped to 'succeeded' inside run_agent_in_background's finally via _quiz_job_update (quiz.py:685-687), and _quiz_job_update SWALLOWS ALL ERRORS (quiz.py:607-608). So the success path where the agent completed (final_result persisted to DB+Redis, job_ok=True) but the mark_succeeded DB write fails (pool exhaustion / transient connection reset on the separate session it opens) leaves the row 'running'. The heartbeat then goes stale (heartbeat() is defined at database.py:525 but never called during the run) and after stale_after_s the sweeper claims and re-runs the job. _recover_one loads the completed state (Redis still has final_result, or _rehydrate_state_from_db rebuilds it including final_result at quiz.py:1742-1743) and re-enters the graph. _phase_router (graph.py:929-938) routes the completed state (ready_for_questions, baseline_ready, answered>=baseline_count) to decide_or_finish, which has NO guard for an existing final_result: it makes a fresh paid decision LLM call (_determine_decision_action) and, on FINISH_NOW, re-invokes the paid tool_write_final_user_profile (graph.py:840-849). _persist_adaptive_and_final then unconditionally mark_completed-overwrites the result and re-runs the paid FAL result-image generation (quiz.py:548-575) with no is_completed check. The module docstring's claim 'Nodes are idempotent: re-running after END will not redo work that exists' (graph.py:19) is false for decide_or_finish. Net effect: duplicate paid LLM+image spend and a silent mutation of the result the user already saw. No test covers this (test_agent_recovery.py only has claim/recover/disabled/no-graph cases).
- **Fix:** Make the recovery path idempotent for finished work. Cheapest: in decide_or_finish, short-circuit when state.get('final_result') is already populated (return {'should_finalize': True, 'final_result': state['final_result']} without any LLM/image call). Defense-in-depth: in _recover_one, skip re-running and mark the job 'succeeded' when the durable SessionHistory row is already is_completed=True (or final_result present); and emit periodic heartbeat() calls during long astream runs so a slow-but-live run is never mistaken for stalled.

#### P1 — Durable quiz_jobs row is created only INSIDE the background task, leaving the highest-probability crash window unprotected
- **Where:** backend/app/api/endpoints/quiz.py:1456 (/proceed add_task), 1613 (/next add_task), 639-640 (mark_running inside run_agent_in_background)
- **Problem:** /proceed and /next schedule run_agent_in_background via background_tasks.add_task WITHOUT first creating a quiz_jobs row. The row is created only when the bg task calls _quiz_job_update(session_id, "running") at line 640 — which executes AFTER the HTTP 202 is returned. If the worker is killed (deploy / OOM / Container Apps scale-in / SIGTERM) in the window between scheduling and the bg task's first DB write, NO quiz_jobs row ever exists. The recovery sweeper claims only rows in status='running', so it has nothing to claim: the quiz is permanently stranded in 'processing' (FE hits the 60s poll timeout → fatal, paid work lost). This is exactly the failure mode the durable-jobs system exists to prevent, and it is unprotected at the single most likely kill instant (right after the response, before the LLM stream starts). The agent_recovery.py docstring's durability guarantee does not hold for this window.
- **Fix:** Create the quiz_jobs row synchronously in the request handler (inside the held session lock for /next, before add_task) via QuizJobRepository.mark_running(quiz_id) + commit, BEFORE scheduling run_agent_in_background. Then run_agent_in_background's mark_running becomes a heartbeat/attempt-bump on an existing row. This guarantees a recoverable 'running' row exists the instant the 202 is sent.

#### P2 — WATCH-conflict-exhausted full-SET fallback can re-clobber concurrent quiz_history/last_served_index for a NON-finalized agent run

#### P2 — claim_stale lacks FOR UPDATE SKIP LOCKED, so two replicas sweeping concurrently can double-claim and re-run the same stalled job
- **Where:** backend/app/services/database.py:546-570 (UPDATE ... WHERE quiz_id IN (SELECT ... LIMIT) updates only last_heartbeat_at, not status)
- **Problem:** The docstring claims the atomic claim makes the sweep 'safe across multiple replicas ... never re-run the same job twice' (database.py:501-502, agent_recovery.py:8-9). But recovery_loop runs in EVERY replica's lifespan (main.py:300), all sweeping on the same ~60s cadence, and claim_stale uses UPDATE quiz_jobs SET last_heartbeat_at=now WHERE quiz_id IN (subquery on status='running' AND last_heartbeat_at<deadline) RETURNING. It does NOT transition status (e.g. to 'recovering') and does NOT use SELECT ... FOR UPDATE SKIP LOCKED. Under Postgres READ COMMITTED, two sweepers whose subquery snapshots both predate either commit can both materialize the same quiz_ids and both UPDATE+RETURN them (the outer per-row recheck only validates IN-membership, not the status/heartbeat predicate), so both replicas call _recover_one and re-run the agent twice. Bounded by max_attempts and the per-session action cap, hence P2, but it directly contradicts the 'never twice' claim and wastes paid LLM on a crashed-job re-run.
- **Fix:** Make the claim genuinely atomic: either transition status to a distinct 'recovering' value in the same UPDATE (so the second sweeper's predicate no longer matches) or build the claim on SELECT quiz_id ... FOR UPDATE SKIP LOCKED then UPDATE the locked rows. Add a concurrency test with two simultaneous claim_stale calls asserting disjoint result sets.

#### P2 — quiz_jobs rows are never cleaned up — one terminal row per quiz session accumulates forever
