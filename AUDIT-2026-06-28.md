## Executive verdict

NOT READY to launch to thousands as-is — there are multiple P0 cost/abuse and infra-misconfiguration blockers that would expose unbounded paid LLM/image spend on day one. The decisive issues: (1) the two endpoints that actually drive the paid agent (/quiz/proceed, /quiz/next) have NO Turnstile and NO per-IP throttle (quiz.py:1158-1244, 1298-1381), while the only rate limiter that does exist keys on a spoofable X-Forwarded-For first hop AND fails open when Redis hiccups (rate_limit.py:89-97, 122-124), so an attacker can drive thousands of paid quizzes with a header and a Redis blip; (2) production effectively runs as environment="local"/debug:true (appconfig.local.yaml:4, config.py:883-927), which collapses nearly every "prod-only" security gate — TrustedHost, the weak-secret startup guard, operator 2FA, and the Turnstile prod path all key off env names the deployment never matches; (3) real secrets in backend/.env are baked into the Docker image (.dockerignore has no exclusion). Beyond security, the system cannot reliably scale: agent work runs as in-process FastAPI BackgroundTasks with no durable queue (quiz.py:478-541), the container runs a single uvicorn worker with no autoscale defined in IaC, and the LLM concurrency cap is per-process so it does not bound provider spend across replicas. These are addressable pre-launch but are not cosmetic — they require a focused hardening sprint (fix env/secret config, gate the paid endpoints, fail-closed limiting with a trusted-proxy IP, a real spend cap, and at minimum a worker/queue topology) before any public traffic; the self-learning and virality gaps can follow shortly after.

The good news: this is a **mature, security-conscious codebase** — token-bucket rate limiting, session locks, Turnstile, SSRF-guard primitives, a cost guard, an evaluator/golden/quarantine apparatus, broad tests, and careful error envelopes all exist. The recurring failure mode is not absence of mechanisms but **mechanisms that are wired but inert, fail-open, or keyed off the wrong environment name** — so the protection looks present in code review yet never executes in the deployed environment (proven live above). Most P0/P1 items are therefore *connect-the-existing-pieces* fixes.

## Cross-cutting themes
*(Root causes; individual findings are downstream of them.)*

**1. Fail-open everything: security and correctness silently disable when Redis hiccups**  
The app's entire defense against the #1 risk (bot/cost abuse) and its concurrency-correctness all depend on Redis, and every one of those checks chooses availability over safety on a Redis error. A single Redis blip (or a deliberately induced one) simultaneously removes per-IP rate limiting, the per-IP quiz-start budget, and the per-session single-flight lock — exactly the window an attacker wants for a cost-spike, and also when double-firing of the paid agent for one session becomes possible. There is no total-spend backstop behind these, so fail-open here means truly unbounded paid spend.
  
_Spans:_ Security (bot/abuse/cost), Reliability (quiz flow). Files: backend/app/security/rate_limit.py:122-124,163-200; backend/app/security/session_lock.py:36-44; backend/app/services/llm_concurrency.py:44-57,140-162; backend/app/core/config.py:253-274

**2. Production is misconfigured as a non-prod environment, so every 'prod-only' hardening gate is dead code**  
A pervasive pattern of env-name-keyed guards combined with a baked appconfig.local.yaml (environment=local, debug=true) and unimplemented Azure App Config means the deployed environment name matches no prod set. The result is a stack of security features that look present in code but never execute: TrustedHostMiddleware, the fail-closed weak-secret startup guard, operator 2FA, and the Turnstile prod path. Compounded by real secrets baked into the Docker image, this is a single root cause that detonates several independent P0/P1 findings at once.
  
_Spans:_ Security (auth/secrets/infra), Architecture. Files: backend/app/core/config.py:470-476,883-927; backend/appconfig.local.yaml:4; backend/.dockerignore:1-15 + backend/Dockerfile:71; backend/app/main.py:307-330; backend/app/api/dependencies.py:350-383; backend/app/services/precompute/secrets.py:60-91

**3. Spoofable client identity (X-Forwarded-For first hop) undermines every per-IP control**  
Client IP is derived from the untrusted first hop of X-Forwarded-For with no trusted-proxy contract, so it is attacker-controlled. Every defense that keys on IP is therefore bypassable by rotating a header: rate limiting, the quiz-start budget, and — most damaging — the flag-based auto-quarantine, where one attacker can quarantine any published pack with five spoofed requests. This turns several layered controls into a single shared weakness.
  
_Spans:_ Security (bot/abuse/cost), Self-learning / self-healing. Files: backend/app/security/rate_limit.py:89-97; backend/app/api/dependencies.py (_client_ip); backend/app/api/endpoints/content.py:61-68,142

**4. Paid work runs as in-process BackgroundTasks with no durable queue — not crash-safe and not scalable**  
The only place paid LLM streaming and image generation happen is FastAPI BackgroundTasks running on the web worker's event loop. Work is lost on any restart/crash, stranding sessions in 'processing' forever (no DB fallback in /status), and it ties cost-bearing compute to request-serving capacity. This is simultaneously a reliability, scalability, and architecture failure, and it is also why no test ever exercises the real paid path. A durable queue/worker tier is the structural fix that several findings point at.
  
_Spans:_ Reliability (quiz flow), Scalability, Architecture, Testing (backend). Files: backend/app/api/endpoints/quiz.py:478-541,1237,1375,1469-1472; backend/Dockerfile:93; backend/app/main.py:159-217; infrastructure/main.bicep:1-29

**5. No total cost ceiling — the #1 risk has no actual budget enforcement anywhere**  
Despite cost-abuse being the stated top risk, there is no global spend cap. The live path has only a per-process concurrency semaphore (which does not bound replicas or total dollars), the precompute cost guard is a pre-check that a single Tier-3 attempt can overshoot and that undercounts re-queued jobs, cost_cents can overflow a SmallInteger at $327.67, and the eval harness has zero token/cost accounting so the team literally cannot measure or optimize spend. The defenses are local and advisory, not a hard limit.
  
_Spans:_ Security (bot/abuse/cost), Scalability, Reliability (precompute), Agent O&E (eval/tuning). Files: backend/app/services/llm_concurrency.py:44-57,140-162; backend/app/services/precompute/cost_guard.py:58-72; backend/app/services/precompute/builder.py:136-155; backend/app/models/db.py:585-587; backend/Analysis/run_experiment.py:97-135

**6. Multiple uncoordinated sources of truth for session state with no reconciliation, plus non-atomic writes that clobber each other**  
Live session state lives in three places (LangGraph checkpointer, CacheRepository/Redis, Postgres) with no reconciliation contract and divergent TTLs (checkpointer unbounded vs cache 3600s). On top of that fragmentation, the background agent's full-state SET and /proceed's save_quiz_state overwrite concurrent atomic merges to quiz_history/last_served_index, and mark_completed overwrites DB history from possibly-stale agent state with swallowed failures. The combination produces lost answers, wedged 'processing' sessions, and data that silently disagrees across stores — hard-to-debug correctness rot under real concurrency.
  
_Spans:_ Architecture, Reliability (quiz flow), Reliability (agent graph). Files: backend/app/api/endpoints/quiz.py:332-349,434-443,1189-1197,1327-1354,1469-1474; backend/app/agent/graph.py:1042-1090; backend/app/services/redis_cache.py:251-330

**7. The self-learning / self-improving promise is unwired: judges, drift probes, and golden-set growth are dead code, so unverified content auto-promotes**  
The product is pitched as self-improving, but the quality loop is structurally absent in production: nightly promotion is structural-only with no content/safety judge, judge_plan_score is never written so the judge gate is dead, the drift probe and golden-set promotion are never invoked by any scheduler, and the golden set never grows from verified data. The net effect is that unverified user-generated quizzes can be signed into prod packs with no accuracy or safety review — a quality-and-safety risk that grows with traffic rather than improving.
  
_Spans:_ Self-learning / self-healing, Agent O&E (eval/tuning). Files: backend/scripts/promote_user_quizzes.py:129-148; backend/app/api/endpoints/admin_precompute.py:490; backend/app/jobs/evaluator_drift_probe.py:30-55; backend/app/services/precompute/golden.py:30-55

**8. No content moderation gate and unmitigated prompt injection before the paid pipeline**  
A moderation/safety classifier exists but is never invoked, so any topic enters the paid LLM+image pipeline with no content gate. Free-text answers and the topic/category are interpolated verbatim into next-question and final-result prompts with no normalization, opening prompt-injection to steer outputs or amplify cost. For an open, no-auth, viral, image-generating app this is both a brand-safety and a cost-amplification exposure at the top of the funnel.
  
_Spans:_ Security (input/injection). Files: backend/app/agent/graph.py:405-522; backend/app/agent/prompts.py:69-89,191-216,312-315; backend/app/api/endpoints/quiz.py:1276-1295; backend/app/agent/tools/content_creation_tools.py:469-482,740-749; backend/app/api/endpoints/content.py:54-82

**9. Launching blind: no analytics/telemetry and broken share/SEO surfaces kill the viral loop**  
There is no analytics or telemetry anywhere, so a launch to thousands would be flown completely blind to funnel, drop-off, and virality. Simultaneously the viral surfaces are broken: shared result links render a generic card (no per-result OG/Twitter meta, no SSR/prerender), og:image/favicon/manifest 404 in production, the CSP blocks Google Fonts, and robots.txt/sitemap are unconfigured placeholders. The growth mechanism the product depends on is both un-instrumented and visibly broken.
  
_Spans:_ Virality (launch readiness), Self-learning / self-healing, UX (flow/modern feel). Files: frontend/src/components/common/ErrorBoundary.tsx:92; frontend/staticwebapp.config.json:2-5,13; frontend/dist/index.html:18-21,31-35; frontend/public/robots.txt:5; frontend/public/sitemap.xml:9

**10. Frontend treats slow-but-normal backend waits as fatal dead-ends, destroying progress mid-quiz**  
The frontend's resilience model is mismatched with backend timing: the poll layer enforces a 60s hard timeout that throws a non-retriable 408 while the backend agent is allowed up to ~90s, so legitimately-slow quizzes strand the user and throw away all progress with only 'Start Over'. Landing dead-ends if Turnstile never resolves, the result-page error state has no action button, and Zod .parse() on every poll misclassifies BE shape drift as a network error. For a viral funnel these convert transient slowness into permanent abandonment.
  
_Spans:_ UX (loading/perceived perf), UX (flow/modern feel), Reliability (frontend). Files: frontend/src/services/apiService.ts:604,662-668; frontend/src/store/quizStore.ts:394-405; frontend/src/pages/LandingPage.tsx:184-240; frontend/src/pages/FinalPage.tsx:167-187

#### [P0] /quiz/proceed and /quiz/next trigger the paid LLM agent with NO Turnstile and NO per-IP throttle
- **Severity:** **P0** &nbsp;|&nbsp; **Dimension:** Security (bot/abuse/cost) &nbsp;|&nbsp; **Status:** REVISE
- **Location:** `backend/app/api/endpoints/quiz.py:1158-1244 (proceed_quiz), 1298-1381 (next_question)`

**Problem.** start_quiz (line 955) is the only quiz endpoint gated by both _enforce_quiz_start_ip_rate_limit (line 968) and verify_turnstile (line 969). proceed_quiz and next_question take only (request, background_tasks, agent_graph, redis_client[, db_session]) — no verify_turnstile, no per-IP limiter. Both schedule run_agent_in_background (quiz.py:1237 and :1375), which runs the full LangGraph agent (question generation, decision-maker, final-profile-writer — all gpt-4o-mini calls per config.py:600-613) plus generate_result_image FAL calls (quiz.py:447-458). #### [P0] Client IP for all rate limits is taken from spoofable X-Forwarded-For first hop, and the limiter fails open
- **Severity:** **P0** &nbsp;|&nbsp; **Dimension:** Security (bot/abuse/cost) &nbsp;|&nbsp; **Status:** CONFIRMED
- **Location:** `backend/app/security/rate_limit.py:89-97 (_client_ip), 122-124 (fail-open)`

**Problem.** _client_ip (rate_limit.py:91-93) returns xff.split(',')[0] — the FIRST hop of X-Forwarded-For, which is fully attacker-controlled unless a trusted proxy overwrites it. The same logic is used by the global rate_limit_middleware (main.py:517), the per-IP quiz-start limiter (quiz.py:931 via _client_ip), and Turnstile remoteip (dependencies.py:200-218). A bot sends a fresh random X-Forwarded-For per request and gets a brand-new token bucket every time, evading BOTH the global per-(IP,prefix) bucket and the per-IP quiz_start bucket entirely. Compounding this, RateLimiter.check fails OPEN on any Redis error (rate_limit.py:122-124 returns allowed=True) and the middleware fails open if redis_factory raises (main.py:511-512). So if Redis is degraded/down, there is literally zero throttling on any endpoint while the LLM+FAL spend continues. No code strips/validates X-Forwarded-For against a configured trusted-proxy count.

**Fix.** Centralize client-IP derivation in one helper that: (1) only honors X-Forwarded-For when request.client.host is in a configured trusted-proxy CIDR set, otherwise uses request.client.host directly; (2) when trusted, parses XFF right-to-left and skips a configured TRUSTED_PROXY_HOPS count, returning the first untrusted address (not the leftmost); (3) validates each candidate is a parseable IP (ipaddress.ip_address) and rejects/normalizes garbage. Replace the three duplicated _client_ip variants (rate_limit.py:89, quiz.py via import, dependencies.py:200) with this helper. For the cost-critical paths, add a cheap in-process fallback limiter (e.g. per-process token bucket keyed by request.client.host) that engages when Redis is unreachable instead of failing fully open, or fail-closed on /quiz/start. Update test_client_ip.py to assert the trusted-hop-skipping behavior.

#### [P1] Per-IP quiz-start budget is too generous and the global bucket gates all /api/quiz/* together, allowing sustained paid-quiz abuse
- **Severity:** **P1** &nbsp;|&nbsp; **Dimension:** Security (bot/abuse/cost) &nbsp;|&nbsp; **Status:** CONFIRMED
- **Location:** `backend/app/core/config.py:253-257 (quiz_start_rate_limit), 264-274 (RateLimitConfig defaults)`

**Problem.** quiz_start_rate_limit defaults to capacity=3, refill 1/30s (config.py:253-256) = ~120 quiz starts/hour from a single (non-spoofed) IP — each start fires a synopsis + character LLM fan-out and schedules synopsis+character FAL image jobs (quiz.py:1130, _schedule_image_jobs_safe). At ~4-6 character images + 1 synopsis image per start (image_pipeline.generate_character_images fans out to all characters), 120 starts/hr is hundreds of paid FAL generations/hr per IP, and that is the INTENDED cap, before any XFF spoofing. ### Architecture

#### [P1] Agent execution runs on FastAPI BackgroundTasks in the web process — no durable queue, work lost on restart/crash
- **Severity:** **P1** &nbsp;|&nbsp; **Dimension:** Architecture &nbsp;|&nbsp; **Status:** CONFIRMED
- **Location:** `backend/app/api/endpoints/quiz.py:1237, 1375, 478-541`

**Problem.** All adaptive question generation and finalization runs via `background_tasks.add_task(run_agent_in_background, ...)` (quiz.py:1237 for /proceed, 1375 for /next), which executes in the same web worker's event loop AFTER the HTTP response is returned. This is the architecture's load-bearing async path for the paid LLM work, yet it has none of the properties a job runner needs:
1. No durability: if the worker is restarted (deploy, OOM, autoscale-down) or crashes between returning 202 and finishing the stream, the run is silently lost. The client keeps polling /quiz/status, which reads CacheRepository (1469) — the state never advances past the last persisted question, so the quiz wedges in 'processing' forever with no retry and no error surfaced.
2. No concurrency bound at this layer: every in-flight /proceed and /next schedules a full graph stream on the web loop; the only backstop is the per-call LLM semaphore. A burst of starts can saturate the event loop with multi-second LLM streams, starving request handling (rate-limit middleware, /status polls) on the same workers.
3. No isolation: a slow/stuck FAL or LLM call in a background task degrades latency for live requests sharing the loop.
The shutdown drain (`_drain_in_flight_work`, main.py:159-217) only waits on the LLM concurrency limiter for `shutdown_grace_s` (15s); a quiz mid-adaptive-loop is abandoned past that window.

**Fix.** Introduce a Redis-backed job record keyed by quiz_id (status/attempts/last_error) written before scheduling and updated in run_agent_in_background's finally; have get_quiz_status surface a "failed"/"stuck" status (e.g. job marked running but no heartbeat past a TTL) so the client stops infinite-polling. Add a startup sweeper that, on boot, scans for jobs in "running" with stale heartbeats and re-enqueues them — since LangGraph state is already checkpointed in Redis under thread_id, resumption is agent_graph.astream(None, config). Longer term, move agent execution off the request event loop into a bounded worker pool or a dedicated consumer process (arq/Dramatiq/Celery) reading a Postgres/Redis job table, so paid LLM streams never share the loop with rate-limit middleware and /status polls, and make the shutdown path wait on the actual background tasks (track them in a set and asyncio.gather with timeout) rather than only the LLM limiter counter.

#### [P1] Flag-based auto-quarantine is trivially abusable: spoofable X-Forwarded-For lets one attacker quarantine any published pack with 5 requests
- **Severity:** **P1** &nbsp;|&nbsp; **Dimension:** Self-learning / self-healing &nbsp;|&nbsp; **Status:** CONFIRMED
- **Location:** `backend/app/api/endpoints/content.py:61-68, 142`

## Coverage gaps (what this audit did NOT fully establish)

- Load/stress and chaos testing: no finding establishes measured throughput, p95 latency, or behavior under the exact failure this audit fears most — Redis down (fail-open) + concurrent quiz-start flood. A real concurrency/race test for /quiz/next is explicitly missing (contention is monkeypatched). The cost-blast-radius is reasoned about but never empirically bounded with a synthetic abuse run.
- Actual cloud/runtime configuration vs. repo config: the audit infers prod runs as env=local from baked appconfig.local.yaml and unimplemented Azure App Config, but the live Container Apps env vars, Key Vault wiring, ingress/proxy (whether Container Apps strips X-Forwarded-For), and the real TRUSTED_HOSTS/CORS values at deploy time were not directly inspected. Confirming the deployed environment name and proxy behavior is essential before trusting the P0 severities.
- Database schema/index/migration health: only a few specific columns are flagged (cost_cents SmallInteger, characters.name UNIQUE). A systematic review of indexes for the hot query paths (precompute lookup adds 3-6 round-trips/start; quarantine cascade loads all rows), Alembic migration reversibility, connection-pool sizing under background fan-out, and N+1 patterns beyond character upsert was not done end-to-end.
- Privacy/compliance and data retention: an open no-auth app still collects topics, free-text answers, feedback comments, and IPs, and bakes user content into promoted prod packs. There is no finding on PII handling, retention/TTL policy for stored quiz history, GDPR/CCPA posture, age-gating, or content-rights for AI-generated images — all relevant when launching publicly to thousands.
- Image-generation cost and safety specifics: FAL spend per quiz is named as the core risk but the actual per-image cost, retry/amplification math, and NSFW/brand-safety filtering on generated images (vs. only topic moderation) are not quantified. The HEAD-probe regeneration and null-never-retried findings hint at waste but the total image-spend envelope is uncharacterized.
- Dependency and supply-chain audit: no review of backend/frontend dependency versions for known CVEs, the LiteLLM/LangGraph/FAL SDK pinning, or lockfile integrity — relevant given secrets-in-image and outbound SSRF concerns already found.
- Observability beyond analytics: the audit flags missing product analytics and over-verbose INFO logging of raw LLM responses, but does not assess backend metrics/tracing/alerting (e.g., spend alarms, queue-depth, error-rate SLOs) needed to even detect a cost-abuse event in real time once launched.
- End-to-end test coverage of the real paid path and recovery: it is noted that run_agent_in_background is never tested and session-lock release is silently swallowed in fixtures, but there is no assessment of whether crash-recovery, restart-mid-quiz, or queue-redelivery behavior could be tested at all given the current in-process design.
